A guard decides *whether* a handler runs. It sees the request before
the handler does and either lets it through (`Ok(())`) or short-circuits
with a typed [`Denial`](https://docs.rs/nest-rs-guards/latest/nest_rs_guards/enum.Denial.html)
the framework maps to a 401 / 403 / 429 response. One trait,
three transports: the same guard impl is reachable from HTTP, GraphQL,
and WS through the [Layer System](/fundamentals/).

`Guard` is shipped by
[`nest-rs-guards`](https://crates.io/crates/nest-rs-guards). It extends
[`Layer`](https://docs.rs/nest-rs-core/latest/nest_rs_core/trait.Layer.html)
so guards plug into the framework's dedup-by-`TypeId` and
declaration-order chain.

## A minimal guard

```rust
use nest_rs_guards::prelude::*;
use nest_rs_http::poem::Request as HttpRequest;

#[injectable]
#[derive(Default)]
pub struct RateLimitGuard;

impl Layer for RateLimitGuard {}

#[async_trait]
impl Guard for RateLimitGuard {
    async fn check_http(&self, req: &mut HttpRequest) -> Result<(), Denial> {
        if over_quota(req) {
            return Err(Denial::rate_limited("rate limited", 60));
        }
        Ok(())
    }
}
```

A guard is an `#[injectable]` provider that implements `Guard`. It
goes in some module's `providers = [...]`, and the access graph
enforces that every controller / resolver / gateway referencing it
imports the right module.

The trait carries one method per transport and inherits `Ok(())`
defaults for the ones the guard doesn't apply to:

```rust
#[async_trait]
pub trait Guard: Layer {
    async fn check_http(&self, _req: &mut HttpRequest) -> Result<(), Denial> { Ok(()) }
    async fn check_graphql(&self, _ctx: &GraphqlContext<'_>) -> Result<(), Denial> { Ok(()) }
    async fn check_ws_message(&self, _client: &WsClient, _event: &str, _data: &Value)
        -> Result<(), Denial> { Ok(()) }
}
```

A no-op default means "doesn't apply to this transport", not "skip
security". Other guards on the same route still run.

## The three scopes

A guard binds at one of three scopes. The container resolves all
three from the same provider — declaring once is enough; the rest is
about *where* the guard runs.

| Scope | Binding | Resolved by |
|-------|---------|-------------|
| **Global** | `App::builder().use_guards_global([guard::<AuthGuard>()])` in `main` | A transport-level interceptor + the per-route shaper |
| **Controller / Resolver / Gateway** | `#[use_guards(AuthGuard, AuthzGuard)]` on the struct | The container, at mount |
| **Per-handler** | `#[use_guards(AdminGuard)]` beside a verb / `#[query]` / `#[subscribe_message]` | The container, at mount |

Multiple guards listed in one attribute run in declaration order,
outermost first — the first listed sees the request before the
second.

```rust
#[controller(path = "/users")]
#[use_guards(AuthGuard, AuthzGuard)]
pub struct UsersController {
    #[inject]
    svc: Arc<UsersService>,
}

#[routes]
impl UsersController {
    #[get("/:id")]
    async fn get(&self, user: Bind<UsersService, Read>) -> Json<User> {
        Json(User::from(&*user))
    }

    #[delete("/:id")]
    #[use_guards(AdminGuard)]
    async fn delete(&self, id: Path<Uuid>) -> Result<()> {
        self.svc.delete(*id).await
    }
}
```

`AdminGuard` runs *on top of* the controller's two guards. Per-handler
is additive, not substitutive. A `#[public]` handler stays open *by
intent* — the absence of guards is the explicit decision.

## Ordering inside a route

Per-route, inner → outer:

```
shaper → interceptors → guards → filters → meta
```

- **Interceptors** wrap the shaper and can install ambient state
  across the guards (`DbContext` installs the SeaORM executor before
  `AbilityGuard` reads from it).
- **Guards** run after interceptors, in declaration order.
- **Meta** is read by guards via `Reflector`.

<Aside>
**Global** interceptors wrap *outside* global guards — the data
context must install the executor around the guards too, so
`AbilityGuard` can read it. Controller- and handler-bound
interceptors wrap *inside* the controller's guards.
</Aside>

## Attach context and route metadata

A guard often does more than gate access — it produces a value the
handler needs (the authenticated principal, the resolved tenant). Attach
on the way in, read on the way out with `Ctx<T>`:

```rust
#[async_trait]
impl Guard for AuthGuard<MyStrategy> {
    async fn check_http(&self, req: &mut HttpRequest) -> Result<(), Denial> {
        let claims = self.strategy.authenticate(req).await?;
        req.extensions_mut().insert(claims);
        Ok(())
    }
}

#[get("/me")]
async fn me(&self, auth: Ctx<Claims>) -> Json<User> {
    Json(self.svc.find(auth.sub).await?)
}
```

`Ctx<T>` rejects with a 500 if the value is absent — a missing
context means the guard that should have set it never ran. Store an
`Arc<_>` if the value is large.

Guards can also read what an earlier guard attached. `AbilityGuard`
reads `Claims` left by `AuthGuard` to build the ambient `Ability` —
which is why declaration order (`#[use_guards(AuthGuard, AuthzGuard)]`)
matters.

For decisions that vary per route — required roles, rate-limit quotas —
`#[meta(...)]` attaches a typed payload; `Reflector` reads it back
inside the guard:

```rust
#[get("/admin")]
#[use_guards(AuthGuard, RolesGuard)]
#[meta(Roles(&["admin", "auditor"]))]
async fn admin(&self) -> &'static str { "ok" }
```

`#[meta(...)]` only reaches guards bound at controller or per-handler
scope. A global guard runs before routing resolves a handler, so the
reflector finds nothing.

`#[public]` uses the same channel. The framework does **not** act on
it — each guard decides what "public" means (`AuthGuard` may still
authenticate when a token is present but not reject anonymous callers;
`AbilityGuard` may still apply visitor rules). Marking a route public
doesn't strip its guards; it tells them to relax.

## Declare on the provider

The Layer System dedups every layer by [`TypeId`](https://doc.rust-lang.org/std/any/struct.TypeId.html)
across the three scopes. When the same guard is declared at several
levels, the **broadest scope wins** and the other declarations log a
`warn` at mount but never re-execute. Redeclaration is free at
runtime.

```rust title="apps/platform-api/src/main.rs"
App::builder()
    .use_guards_global([guard::<AuthGuard>(), guard::<AuthzGuard>()])
    .module::<AppModule>()
```

```rust title="crates/features/src/users/http/controller.rs"
#[controller(path = "/users")]
#[use_guards(AuthGuard, AuthzGuard)]
pub struct UsersController { /* ... */ }
```

`AuthGuard` and `AuthzGuard` run **exactly once** per request, not
twice. The cost of the controller's two extra lines is zero runtime
overhead and one `warn` at boot.

**Declare layers on the provider that needs them, not only at the app
boundary.** A `crates/features` controller is designed to be portable —
if it relies only on `use_guards_global([AuthGuard, AuthzGuard])` to
be secure, forgetting that line in a second app silently turns every
route public. The compiler will not complain. Redeclaring on the
controller binds the security policy to the code that needs it. The
same rule applies to interceptors and filters.

Within one scope, guards run in the order they appear in
`#[use_guards(...)]`. Across scopes, the chain composes global →
controller → handler, outermost first. `Layer::priority` is an
optional tiebreaker when declaration order can't express the intent —
most guards leave it at `0`.

## HTTP, GraphQL, and WebSockets

For most apps the global HTTP declaration is enough: the GraphQL
`POST /graphql` and the WS upgrade are both HTTP requests that pass
through the global chain.

Each transport method runs at a different seam:

- `check_http` — on `&mut HttpRequest` before routing. Gate and mutate:
  attach `Claims`, install per-request state in extensions.
- `check_graphql` — inside the resolver's per-operation chain. The
  context is read-only, so a GraphQL guard reads what an upstream
  operation guard seeded.
- `check_ws_message` — once per inbound message, after the connection
  upgrade.

GraphQL and WS need a bridge one level above the resolver / gateway
because the per-operation context doesn't carry the poem request:

- GraphQL — a `GraphqlOperationGuard` (e.g. `GraphqlAbilityBridge`)
  re-runs the HTTP guard chain on `POST /graphql` and seeds `Ability`
  for the operation. Per-resolver gating uses `GraphqlResolverGuard`.
- WS — a `SocketContext` (e.g. `WsDataContext`) captures pool +
  `Ability` on upgrade and re-installs them around each message.
  Per-message gating goes through `check_ws_message`.

On a resolver or gateway, `#[use_guards(...)]` often carries a
**marker guard** — a struct whose job is to declare a dependency on
the seeded context so the access graph can validate it. Omitting the
matching authz module fails boot with a clear error naming the
missing guard.

```rust
#[resolver]
#[use_guards(GraphqlAuthGuard)]
impl UsersResolver { /* ... */ }
```

```rust
#[gateway(path = "/chat")]
#[use_guards(AuthGuard, AuthzGuard)]
pub struct ChatGateway { /* ... */ }

#[messages]
impl ChatGateway {
    #[subscribe_message("send")]
    #[use_guards(WsAuthGuard)]
    async fn send(&self, msg: SendMessage) -> Reply { /* ... */ }
}
```

for the full HTTP / GraphQL / WS wiring, and
[WebSockets / Guards](/websockets/guards/) for the two WS scopes
(connection vs message).

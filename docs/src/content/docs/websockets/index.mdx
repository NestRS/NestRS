A WebSocket upgrade is an HTTP `GET`, so a gateway **self-mounts** on the
existing `HttpTransport` — same port, same CORS, same TLS, no second
server. Every frame rides one JSON envelope, `{ "event": "...", "data":
... }`, and each `#[subscribe_message("event")]` method handles one
event. List the gateway in `#[module(providers = [...])]` and you have
real-time wired in.

`nest-rs-ws` is a thin layer over [`poem`'s WebSocket
support](https://docs.rs/poem/latest/poem/web/websocket/) — the same
transport `nest-rs-http` already uses. It hides the read/write split,
the outbox writer task, the connection registry, and the dispatch
plumbing behind one decorator.

## A one-event gateway

The smallest possible gateway: one event, one reply.

```rust title="apps/hello-ws/src/gateway.rs"
use nest_rs_ws::{gateway, messages};

#[gateway(path = "/ws")]
#[derive(Default)]
pub struct HelloGateway;

#[messages]
impl HelloGateway {
    #[subscribe_message("ping")]
    async fn ping(&self) -> &'static str {
        "pong"
    }
}
```

`#[gateway(path = "/ws")]` mounts on the HTTP transport at `/ws`.
`#[messages]` turns the impl block into a dispatcher: the read loop
parses each frame's envelope, matches the event, deserializes `data`,
calls the handler, and serializes the return into a reply frame on the
same event name.

Wire it in like any other provider:

```rust title="apps/hello-ws/src/module.rs"
use nest_rs_core::module;
use nest_rs_http::HttpModule;
use nest_rs_ws::WsModule;

use crate::gateway::HelloGateway;

#[module(
    imports = [HttpModule::for_root(None), WsModule],
    providers = [HelloGateway],
)]
pub struct AppModule;
```

`HttpModule` for the transport, `WsModule` for the connection registry
(`WsServer`). The gateway itself is listed in `providers` — no
`.transport(...)` call, no second port.

## Run it

```bash
just dev hello-ws
```

```text frame="terminal"
2026-06-08T10:23:14Z  INFO nest_rs::http: bound 1 gateway on 0.0.0.0:3000
  GET /ws -> HelloGateway (1 message, 0 lifecycle hooks)
```

Drive it from any WebSocket client. With
[`websocat`](https://github.com/vi/websocat):

```bash frame="terminal"
$ websocat ws://localhost:3000/ws
{"event":"ping","data":null}
{"event":"ping","data":"pong"}
```

The envelope you send is `{event, data}`; the server's reply rides the
same shape on the same event name. From here, every other concern is a
matter of adding one decorator.

## Where to go next

- [Messages](/websockets/messages/) — the envelope shape, the
  `Result`/`Err` return contract, payload validation, error frames.
- [Rooms](/websockets/rooms/) — `WsClient::join` / `leave` /
  `broadcast` / `to`, the per-connection handle.
- [Server-side push](/websockets/server-push/) — `WsServer<N>` from
  outside a handler (queue processors, scheduled jobs, HTTP routes).
- [Namespaces](/websockets/namespaces/) — multiple gateways with
  isolated registries via `WsServer<N>` markers.
- [Guards](/websockets/guards/) — authenticate at the upgrade, gate
  per-message envelopes, the connection-vs-message scope split.
- [Security](/security/) — wiring `AuthGuard` and `AuthzGuard` for WS
  the same way as HTTP and GraphQL.

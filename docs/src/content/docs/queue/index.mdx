A queue handler is **a method** on any `#[injectable]` provider, tagged with
`#[process(queue = "...", concurrency, retries)]` inside a `#[processor]`
impl block. Importing `QueueWorkerModule` in the worker app's `AppModule`
attaches the worker runtime at boot.

The abstractions — `Job`, `Processor`, `ProcessMethod`, the `#[processor]`
macro — ship in **`nest-rs-queue`**. The first-class storage, **Redis**,
ships in **`nest-rs-redis`** (built on [`apalis`](https://docs.rs/apalis)):
the `QueueConnection` producer, the `QueueWorker` transport, and the
`QueueModule` / `QueueWorkerModule` activation seams. An app depends on
both: `nest-rs-queue` for the macro + abstractions, `nest-rs-redis` for the
Redis-bound types.

## A first queue handler

A regular `#[injectable]` service with one decorated method. The
`#[processor]` attribute marks the impl block; `#[process]` marks the method
as a queue consumer.

```rust title="crates/features/src/audio/queue/processor.rs"
use std::sync::Arc;

use anyhow::Result;
use nest_rs_core::injectable;
use nest_rs_queue::processor;

use crate::audio::{AudioService, TranscodeJob};

#[injectable]
pub struct AudioJobs {
    #[inject]
    svc: Arc<AudioService>,
}

#[processor]
impl AudioJobs {
    #[process(queue = "audio", concurrency = 5, retries = 3)]
    async fn transcode(&self, job: TranscodeJob) -> Result<()> {
        self.svc.transcode(&job.file).await
    }
}
```

The processor is **thin**: it owns no business logic of its own — it
hands `job.file` to `AudioService::transcode` and propagates the result.
The same `AudioService` is what the HTTP producer below calls to
enqueue, so a single audited choke point describes everything the
feature can do. Adapters carry transport mechanics; the service carries
intent.

- **`#[injectable]`** makes `AudioJobs` a regular DI provider.
- **`#[processor]`** on the impl block orchestrates the per-method
  `#[process]` attributes and submits one queue consumer per decorated
  method.
- **`#[process(queue, concurrency, retries)]`** binds the method to a
  Redis queue name (stringly-typed, the known cost), bounds the in-flight
  jobs, and gives an apalis retry budget. The job type is read from the
  method's second parameter.

## Several methods on the same provider

The pattern the framework is built for: one `#[injectable]` declares the
deps once, multiple decorated methods share them. Different queues,
different concurrencies, same service:

```rust title="crates/features/src/audio/queue/processor.rs"
#[injectable]
pub struct AudioJobs {
    #[inject] svc_audio: Arc<AudioService>,
    #[inject] svc_media: Arc<MediaService>,
}

#[processor]
impl AudioJobs {
    #[process(queue = "audio", concurrency = 5, retries = 3)]
    async fn transcode(&self, job: TranscodeJob) -> Result<()> {
        self.svc_audio.transcode(&job.file).await
    }

    #[process(queue = "audio.preview", concurrency = 10, retries = 1)]
    async fn preview(&self, job: PreviewJob) -> Result<()> {
        self.svc_media.preview(&job.file).await
    }

    #[process(queue = "audio.cleanup")]
    async fn cleanup(&self, job: CleanupJob) -> Result<()> {
        self.svc_media.cleanup(job.older_than).await
    }
}
```

`#[process(queue = "audio.cleanup")]` omits the optional `concurrency`
(default `1`) and `retries` (default `0`).

Three inventory entries — `AudioJobs::transcode`, `AudioJobs::preview`,
`AudioJobs::cleanup` — all pointing at the same `AudioJobs` instance.
Same `Arc<AudioService>`, same `Arc<MediaService>`, three queues. Each
method still delegates: the processor is a router, never a place where
business logic accumulates.

<Aside>
A single `#[processor]` impl block per struct (multiple would conflict at
the discovery layer). Put every queue handler of a service in the same
block — same convention as `#[routes]` and `#[scheduled]`.
</Aside>

## The payload

```rust title="crates/features/src/audio/dto.rs"
use schemars::JsonSchema;
use serde::{Deserialize, Serialize};

pub const AUDIO_QUEUE: &str = "audio";

// JsonSchema too: the same payload is the producer endpoint's HTTP body.
#[derive(Clone, Debug, Deserialize, Serialize, JsonSchema)]
pub struct TranscodeJob {
    pub file: String,
}
```

A job is **any** `Serialize + DeserializeOwned + Send + 'static` type — no
special trait. The convention is to keep payloads small (just a reference
to the work, not the work itself).

## A producer

A producer is any code that asks the **service** to enqueue. The HTTP
controller is a thin transport adapter: it accepts the body, calls the
service, returns the wire response. It does **not** hold
`Arc<QueueConnection>` and does **not** know the queue name — those are
internal to `AudioService`.

```rust title="crates/features/src/audio/service.rs"
use std::sync::Arc;

use anyhow::Result;
use nest_rs_core::injectable;
use nest_rs_redis::QueueConnection;

use super::dto::{AUDIO_QUEUE, TranscodeJob};

#[injectable]
pub struct AudioService {
    #[inject]
    queue: Arc<QueueConnection>,
}

impl AudioService {
    pub async fn enqueue_transcode(&self, file: String) -> Result<()> {
        self.queue
            .of::<TranscodeJob>(AUDIO_QUEUE)
            .push(TranscodeJob { file: file.clone() })
            .await?;
        tracing::info!(target: "features::audio", %file, "enqueued transcode job");
        Ok(())
    }

    pub async fn transcode(&self, file: &str) -> Result<()> {
        // … the consumer-side work
        Ok(())
    }
}
```

```rust title="crates/features/src/audio/http/controller.rs"
use std::sync::Arc;

use nest_rs_http::{controller, routes};
use poem::http::StatusCode;
use poem::web::Json;
use poem::{Error, Result};

use crate::audio::{AudioService, TranscodeJob};

#[controller(path = "/audio")]
#[use_guards(AuthGuard, AuthzGuard)]
pub struct AudioController {
    #[inject]
    svc: Arc<AudioService>,
}

#[routes]
impl AudioController {
    #[post("/transcode")]
    async fn transcode(&self, body: Json<TranscodeJob>) -> Result<Json<TranscodeJob>> {
        let job = body.0;
        self.svc
            .enqueue_transcode(job.file.clone())
            .await
            .map_err(|e| Error::from_string(e.to_string(), StatusCode::INTERNAL_SERVER_ERROR))?;
        Ok(Json(job))
    }
}
```

`queue.of::<T>(name)` returns a typed publisher bound to the queue name
— and it lives **inside the service**, the single place that knows
`AUDIO_QUEUE`. The producer and consumer share the payload type by
importing it from the same crate; the queue name is the only
stringly-typed coupling, and it stays private to `AudioService`.

## Wire it in

Two modules, two roles:

- **`QueueModule::for_root(None)`** — provides the shared `QueueConnection`.
  Every app (producer *and* consumer) needs it.
- **`QueueWorkerModule`** — activates the consumer runtime: at boot the
  framework drains the discovered `#[process]` methods, spawns one apalis
  worker per method, and shuts them down with the app. A producer-only
  app does NOT import this — keeping its binary free of a worker it
  doesn't run.

```rust title="apps/platform-worker/src/module.rs"
use nest_rs_core::module;
use nest_rs_redis::{QueueModule, QueueWorkerModule};

use features::audio::{AudioModule, AudioQueueModule};

#[module(imports = [
    ConfigModule::for_root(),
    QueueModule::for_root(None),
    QueueWorkerModule,
    AudioModule,
    AudioQueueModule,
])]
pub struct PlatformWorkerModule;
```

And `main` is the universal skeleton:

```rust title="apps/platform-worker/src/main.rs"
use anyhow::Result;
use nest_rs_core::App;

use platform_worker::PlatformWorkerModule;

#[tokio::main]
async fn main() -> Result<()> {
    App::builder()
        .module::<PlatformWorkerModule>()
        .build()
        .await?
        .run()
        .await
}
```

That is exactly what `apps/platform-api` is: a producer-only binary. It
imports `QueueModule::for_root` (plus the audio feature's `http` and
`schedule` edges) to gain `QueueConnection` and **push**, but **never
imports `QueueWorkerModule`** — it runs no processors.

## Retries and failure

```rust
#[process(queue = "audio", concurrency = 5, retries = 3)]
```

A method that returns `Err` is **retried** by apalis with backoff up to
`retries` times. After exhaustion, the job is moved to the queue's failed
list (apalis-specific). A panic in the method is treated as a failure.

## The ambient data context

Like a scheduled method, a `#[process]` handler runs as **system work**:

- A **pool executor** is installed (importing `DatabaseModule` is
  enough); `Repo` reads go through the pool.
- **No ability** is installed; `Repo` reads are **unscoped** — correct
  for system work.

## Module-gated, even when the crate is shared

The worker depends on the shared `features` crate, so the audio
controller's code is *compiled in* — but **module-gating** keeps every
provider the worker doesn't import inert. Importing only
`AudioModule + AudioQueueModule`, the HTTP controller, the OpenAPI
doc, and the GraphQL schema never mount: present in the binary, dead at
runtime.

A queue is the durable, distributed counterpart to a scheduled job. A
**producer** pushes a typed payload onto a named queue; a
**consumer** (a `#[processor]`) reads from the same name with its own
concurrency and retry policy. They are decoupled — different binaries,
different deploys, different replica counts.

Backed by Redis through [`apalis`](https://docs.rs/apalis), wrapped by
the `nestrs-queue` crate so the framework owns the DI shape and the
discovery registry.

## A durable consumer

This is the real consumer from `apps/platform-worker`:

```rust title="crates/features/src/audio/queue/processor.rs"
use std::sync::Arc;

use anyhow::Result;
use nestrs_queue::{async_trait, processor, Processor};

use crate::audio::core::{Transcoder, TranscodeJob};

#[processor(queue = "audio", concurrency = 5, retries = 3)]
pub struct AudioProcessor {
    #[inject]
    transcoder: Arc<Transcoder>,
}

#[async_trait]
impl Processor for AudioProcessor {
    type Job = TranscodeJob;

    async fn process(&self, job: TranscodeJob) -> Result<()> {
        self.transcoder.transcode(&job.file).await
    }
}
```

- **`#[processor(queue, concurrency, retries)]`** — `queue` is the
  Redis queue name (stringly-typed, the known cost — there is no
  separately-typed `Queue<TranscodeJob>`). `concurrency` caps the
  in-flight jobs per worker. `retries` is the upper bound on retry
  attempts (apalis backoff applies).
- **`type Job = TranscodeJob`** — the typed payload. The framework
  serdes the job at the queue boundary; `TranscodeJob` is what your
  `process` method receives.

## The payload

```rust title="crates/features/src/audio/core/dto.rs"
use schemars::JsonSchema;
use serde::{Deserialize, Serialize};

pub const AUDIO_QUEUE: &str = "audio";

// JsonSchema too: the same payload is the producer endpoint's HTTP body.
#[derive(Clone, Debug, Deserialize, Serialize, JsonSchema)]
pub struct TranscodeJob {
    pub file: String,
}
```

A job is **any** `Serialize + DeserializeOwned + Send + 'static`
type — no special trait. The convention is to keep payloads small
(just a reference to the work, not the work itself).

## A producer

A producer is any code holding `Arc<QueueConnection>`. Here it's the HTTP
handler the `api` app exposes — a request enqueues a job the separate
`worker` app consumes:

```rust title="crates/features/src/audio/http/controller.rs"
use std::sync::Arc;

use nestrs_http::{controller, routes};
use nestrs_queue::QueueConnection;
use poem::http::StatusCode;
use poem::web::Json;
use poem::{Error, Result};

use crate::audio::core::{TranscodeJob, AUDIO_QUEUE};

#[controller(path = "/audio")]
#[use_guards(AuthGuard, AppAbilityGuard)]
pub struct AudioController {
    #[inject]
    queue: Arc<QueueConnection>,
}

#[routes]
impl AudioController {
    #[post("/transcode")]
    async fn transcode(&self, body: Json<TranscodeJob>) -> Result<Json<TranscodeJob>> {
        let job = body.0;
        self.queue
            .of::<TranscodeJob>(AUDIO_QUEUE)
            .push(job.clone())
            .await
            .map_err(|e| Error::from_string(e.to_string(), StatusCode::INTERNAL_SERVER_ERROR))?;
        Ok(Json(job))
    }
}
```

`queue.of::<T>(name)` returns a typed publisher bound to the queue
name. The producer and consumer share the type by importing it from
the same crate — the queue name is the only stringly-typed coupling.

## Wire it in

The consumer side runs in its own binary (or alongside others). Mount
the `QueueWorker` transport in `main` and import
`QueueModule::for_root`:

```rust title="apps/platform-worker/src/app.rs"
use nestrs_core::module;
use nestrs_queue::QueueModule;
use features::audio::{AudioCoreModule, AudioQueueModule};

#[module(
    imports = [
        QueueModule::for_root(None),     // reads NESTRS_QUEUE__URL
        AudioCoreModule,
        AudioQueueModule,                // the #[processor]
    ],
)]
pub struct AppModule;
```

```rust title="apps/platform-worker/src/main.rs"
use nestrs_core::App;
use nestrs_queue::QueueWorker;
use platform_worker::AppModule;

#[tokio::main]
async fn main() -> anyhow::Result<()> {
    App::builder()
        .module::<AppModule>()
        .build()
        .await?
        .transport(QueueWorker::new())       // ← drains the inventory of processors
        .run()
        .await
}
```

That is exactly what `apps/platform-api` is: a producer-only binary. It
imports `QueueModule::for_root` (plus the audio feature's `http` and
`schedule` edges) to gain `QueueConnection` and **push**, but never mounts
`QueueWorker` — it runs no processors.

Here producer and consumer are **split across deploys**: `apps/platform-api`
produces (an HTTP handler plus a `#[cron_job]`, mounting `Scheduler`), and
`apps/platform-worker` consumes (`QueueWorker`). The Redis queue and the
shared `features::audio` contract are all they share — no RPC. They could
just as well run in one binary; the queue doesn't care.

## Retries and failure

```rust
#[processor(queue = "audio", concurrency = 5, retries = 3)]
```

A processor that returns `Err` is **retried** by apalis with backoff
up to `retries` times. After exhaustion, the job is moved to the
queue's failed list (apalis-specific). A panic in `process` is
treated as a failure.

## The ambient data context

Like a scheduled job, a processor runs as **system work**:

- A **pool executor** is installed (importing `DatabaseModule` is
  enough); `Repo` reads go through the pool.
- **No ability** is installed; `Repo` reads are **unscoped** — correct
  for system work.

## Module-gated, even when the crate is shared

The worker depends on the shared `features` crate, so the audio
controller's code is *compiled in* — but **module-gated discovery** keeps
every provider the worker doesn't import inert. Importing only
`AudioCoreModule + AudioQueueModule`, the HTTP controller, the OpenAPI
doc, and the GraphQL schema never mount: present in the binary, dead at
runtime. (Trimming them from the binary itself — cargo features on
`features` to drop the unused transports — is a build-time concern on the
roadmap.)

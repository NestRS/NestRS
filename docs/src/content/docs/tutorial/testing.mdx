You ship one e2e test that boots the *real* `TutorialApiModule`
against a throwaway Postgres, drives it through the same poem runtime
production uses, and asserts on the round-trip you've been verifying
by hand. By the end of this page, `just test-e2e` runs it green, a
wiring regression in any of the previous seven steps fails this test,
and the feature is locked down.

## One test file per app

Every app ships one `tests/e2e.rs`. Cargo compiles it as a separate
binary; nextest gates it through a binary filter, not `#[ignore]`.

The crate driving this page is
[**`nest-rs-testing`**](https://github.com/NestRS/NestRS/tree/main/crates/nest-rs-testing) —
it provides `TestApp` (the in-process boot harness) and
`EphemeralDatabase` (a throwaway Postgres provisioned through
[testcontainers](https://docs.rs/testcontainers)).

<FileTree>
- apps/tutorial-api/
  - tests/
    - e2e.rs
</FileTree>

Add the dev-dependencies the test needs:

```toml title="apps/tutorial-api/Cargo.toml"
[dev-dependencies]
nest-rs-testing = { workspace = true, features = ["orm"] }
migrations = { workspace = true }
poem = { workspace = true, features = ["test"] }
serde_json.workspace = true
uuid.workspace = true
```

## The boot harness

`TestApp::builder()` runs the same four-phase boot as `main`,
including the access-graph check. The harness seeds the connection so
the module never reaches a real network.

```rust title="apps/tutorial-api/tests/e2e.rs"
use features::{Claims, Role};
use nest_rs_authn::{JwtConfig, JwtOptions, JwtService};
use nest_rs_testing::{EphemeralDatabase, TestApp};
use poem::http::{StatusCode, header};
use serde_json::json;
use uuid::Uuid;

use tutorial_api::TutorialApiModule;

const DEV_PRIVATE_KEY: &str = "-----BEGIN PRIVATE KEY-----\nMC4CAQAwBQYDK2VwBCIEIEYTRN4vmCuIfaUslO5G9pKyxkDJn3q3t9WDHo2FCfw3\n-----END PRIVATE KEY-----\n";
const DEV_PUBLIC_KEY: &str = "-----BEGIN PUBLIC KEY-----\nMCowBQYDK2VwAyEAHfPOjd2Y3m1BLM5nBJBMZFAlfWt69WL1NY8XyYeGfeo=\n-----END PUBLIC KEY-----\n";

async fn boot() -> (EphemeralDatabase, TestApp) {
    let db = EphemeralDatabase::create::<migrations::Migrator>()
        .await
        .expect("create + migrate a throwaway database");
    let app = TestApp::builder()
        .module::<TutorialApiModule>()
        .with_test_telemetry()
        .provide_arc(db.connection())
        .provide(JwtConfig {
            public_key: Some(DEV_PUBLIC_KEY.into()),
            ..Default::default()
        })
        .build()
        .await
        .expect("TutorialApiModule boots against the throwaway database");
    (db, app)
}

fn admin_token() -> String {
    let jwt = JwtService::new(JwtOptions::eddsa(DEV_PRIVATE_KEY, DEV_PUBLIC_KEY))
        .expect("the dev keypair parses");
    jwt.sign(&Claims {
        sub: None,
        org_id: Uuid::now_v7(),
        roles: vec![Role::Admin],
        exp: jwt.expiry(),
    })
    .expect("sign the test token")
}
```

The dev keypair is committed in the test file on purpose — a public
keypair with no value outside the test binary, no secret to leak.
Production reads the keys from `NESTRS_AUTHN__JWT__*` env vars.

You need a `tutorial-api` library target to import `TutorialApiModule`
from the test binary. Add the lib alongside `main.rs`:

```rust title="apps/tutorial-api/src/lib.rs"
mod module;
pub use module::TutorialApiModule;
```

```toml title="apps/tutorial-api/Cargo.toml"
[lib]
path = "src/lib.rs"

[[bin]]
name = "tutorial-api"
path = "src/main.rs"
```

## The round-trip test

The shape of an e2e test: build the app, hit the route, read the
response. Two requests, three assertions — enough to cover the entire
wiring chain.

```rust title="apps/tutorial-api/tests/e2e.rs"
#[tokio::test]
async fn users_round_trip_under_a_bearer_token() {
    let (_db, app) = boot().await;
    let bearer = format!("Bearer {}", admin_token());

    let created = app
        .http()
        .post("/users")
        .header(header::AUTHORIZATION, &bearer)
        .body_json(&json!({ "name": "Ada", "email": "ada@acme.test" }))
        .send()
        .await;
    created.assert_status_is_ok();
    let body = created.json().await;
    let id = body.value().object().get("id").string().to_owned();

    let listed = app
        .http()
        .get("/users")
        .header(header::AUTHORIZATION, &bearer)
        .send()
        .await;
    listed.assert_status_is_ok();
    let names: Vec<String> = listed
        .json()
        .await
        .value()
        .array()
        .iter()
        .map(|u| u.object().get("name").string().to_owned())
        .collect();
    assert_eq!(names, vec!["Ada".to_string()]);

    let got = app
        .http()
        .get(format!("/users/{id}"))
        .header(header::AUTHORIZATION, &bearer)
        .send()
        .await;
    got.assert_status_is_ok();
    assert_eq!(got.json().await.value().object().get("name").string(), "Ada");
}
```

The three assertions cover the entire chain: the `POST` walks
validation, authn, authz, the transaction interceptor, and the
service; the `GET /users` walks the row-level filter and the response
masker; the `GET /users/:id` walks `Bind`'s service load.

## The four failure modes

A happy-path test is half a test. The bugs that ship live in the
failures. Add four short ones — exactly the matrix the
[negative-path testing page](/testing/negative-path/) recommends.

```rust title="apps/tutorial-api/tests/e2e.rs"
#[tokio::test]
async fn create_without_a_token_returns_401() {
    let (_db, app) = boot().await;
    app.http()
        .post("/users")
        .body_json(&json!({ "name": "Ada", "email": "ada@acme.test" }))
        .send()
        .await
        .assert_status(StatusCode::UNAUTHORIZED);
}

#[tokio::test]
async fn create_with_invalid_email_returns_400() {
    let (_db, app) = boot().await;
    let bearer = format!("Bearer {}", admin_token());
    let resp = app
        .http()
        .post("/users")
        .header(header::AUTHORIZATION, &bearer)
        .body_json(&json!({ "name": "Ada", "email": "not-an-email" }))
        .send()
        .await;
    resp.assert_status(StatusCode::BAD_REQUEST);
    let body = resp.json().await;
    assert!(body.value().object().get("details").object().get_opt("email").is_some());
}

#[tokio::test]
async fn cross_tenant_get_returns_403() {
    let (_db, app) = boot().await;
    let bearer_a = format!("Bearer {}", admin_token());
    let bearer_b = format!("Bearer {}", admin_token());

    let id = app
        .http()
        .post("/users")
        .header(header::AUTHORIZATION, &bearer_a)
        .body_json(&json!({ "name": "Ada", "email": "ada@acme.test" }))
        .send()
        .await
        .json()
        .await
        .value()
        .object()
        .get("id")
        .string()
        .to_owned();

    app.http()
        .get(format!("/users/{id}"))
        .header(header::AUTHORIZATION, &bearer_b)
        .send()
        .await
        .assert_status(StatusCode::FORBIDDEN);
}

#[tokio::test]
async fn duplicate_email_does_not_persist() {
    let (_db, app) = boot().await;
    let bearer = format!("Bearer {}", admin_token());

    app.http()
        .post("/users")
        .header(header::AUTHORIZATION, &bearer)
        .body_json(&json!({ "name": "Ada", "email": "dup@acme.test" }))
        .send()
        .await
        .assert_status_is_ok();

    app.http()
        .post("/users")
        .header(header::AUTHORIZATION, &bearer)
        .body_json(&json!({ "name": "Grace", "email": "dup@acme.test" }))
        .send()
        .await
        .assert_status(StatusCode::INTERNAL_SERVER_ERROR);

    let listed = app
        .http()
        .get("/users")
        .header(header::AUTHORIZATION, &bearer)
        .send()
        .await;
    let names: Vec<String> = listed
        .json()
        .await
        .value()
        .array()
        .iter()
        .map(|u| u.object().get("name").string().to_owned())
        .collect();
    assert_eq!(names, vec!["Ada".to_string()], "the failed insert rolled back");
}
```

The duplicate-email test is the most load-bearing of the four: it
proves the transaction interceptor unwinds on a 5xx response. A
regression that drops the transaction wrapper would let the second
insert commit; the assertion catches it.

<Aside type="tip">
Each `admin_token()` call generates a fresh org id — a cross-tenant
test gets two distinct orgs for free. When the same caller needs the
same org across two tokens, factor out an `Uuid::now_v7()` and pass it
into both.
</Aside>

## Run it

`just test-e2e` is the recipe; it routes to nextest with the binary
filter for the e2e file.

<Steps>

1. Bring Postgres up (already done in page 6).

   ```bash frame="terminal"
   $ just db up
   postgres ready on 127.0.0.1:5432
   ```

2. Run the e2e suite.

   ```bash frame="terminal"
   $ just test-e2e
   ────────────
        Starting 5 tests across 1 binary
         PASS [   1.4s] tutorial-api::e2e users_round_trip_under_a_bearer_token
         PASS [   0.8s] tutorial-api::e2e create_without_a_token_returns_401
         PASS [   0.8s] tutorial-api::e2e create_with_invalid_email_returns_400
         PASS [   1.1s] tutorial-api::e2e cross_tenant_get_returns_403
         PASS [   1.0s] tutorial-api::e2e duplicate_email_does_not_persist
   ```

3. Run the rest of the workspace too — unit and integration tests
   don't need a database.

   ```bash frame="terminal"
   $ just test
   ```

</Steps>

`just test` and `just test-e2e` are the only two recipes you need; the
third, `just test-cov`, exists for coverage runs. None of them use
`#[ignore]` — gating is the nextest binary filter on `tests/e2e.rs`.

## What you have now

- A `tests/e2e.rs` booting the real `TutorialApiModule` against a
  throwaway Postgres provisioned by testcontainers.
- A happy-path test and four negative-path tests covering authn,
  validation, cross-tenant access, and transaction rollback.
- A `just test-e2e` recipe running them green — a wiring regression
  in any of the previous seven pages fails here, in this binary.

## Where you are now

You've built `users` end to end: an entity declared once, a service
that gates every DB call, an HTTP controller and a GraphQL resolver
sharing the same policy, a transactional Postgres, an e2e suite that
locks the shape down. The feature lives in `crates/features/src/users/`;
copying it is the way to add the next one.

The reference workspace's `users` (and `orgs`, `audio`, `oauth`) live
in the same shape — open them when a question about a corner case
beats reading more docs.

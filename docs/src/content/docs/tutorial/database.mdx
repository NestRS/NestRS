You add one module to the app root and one migration to the workspace,
and the data layer comes alive. By the end of this page, `POST /users`
persists into Postgres inside a transaction, a failing handler rolls
back automatically, and the e2e baseline you're about to add (next
page) has a real database to point at.

## The data layer in one import

`DatabaseModule::for_root(None)` activates the SeaORM data layer:
seeds the connection from `NESTRS_DATABASE__URL`, registers the
transaction interceptor, binds the worker context. The configuration
uses the framework's dual-path rule — pin a value here or leave it
`None` and let env vars supply it.

The data layer crates are
[**`nest-rs-database`**](https://github.com/NestRS/NestRS/tree/main/crates/nest-rs-database)
(the store-agnostic seam) and
[**`nest-rs-seaorm`**](https://github.com/NestRS/NestRS/tree/main/crates/nest-rs-seaorm)
(the SeaORM implementation, on top of [SeaORM](https://www.sea-ql.org/SeaORM/)).

```rust title="apps/tutorial-api/src/module.rs" {7, 14}
use nest_rs_core::module;
use nest_rs_http::HttpModule;
use nest_rs_seaorm::DatabaseModule;

use features::authn::AuthnModule;
use features::authz::AuthzHttpModule;
use features::users::UsersHttpModule;

#[module(
    imports = [
        HttpModule::for_root(None),
        DatabaseModule::for_root(None),
        AuthnModule,
        AuthzHttpModule,
        UsersHttpModule,
    ],
)]
pub struct TutorialApiModule;
```

Add the crate to the app's `Cargo.toml`:

```toml title="apps/tutorial-api/Cargo.toml"
nest-rs-seaorm = { workspace = true, features = ["http"] }
```

The `http` feature flag turns on the `Bind<S, A>` extractor and the
response masking interceptor — exactly what the controller from page 5
needs.

## A migration for the users table

The workspace ships a single `migrations` crate that holds every
SeaORM migration. Add the `user` table.

<FileTree>
- crates/migrations/
  - src/
    - lib.rs
    - m20260601_000010_create_user.rs
</FileTree>

```rust title="crates/migrations/src/m20260601_000010_create_user.rs"
use sea_orm_migration::prelude::*;

#[derive(DeriveMigrationName)]
pub struct Migration;

#[derive(DeriveIden)]
enum User {
    Table,
    Id,
    OrgId,
    Name,
    Email,
}

#[async_trait::async_trait]
impl MigrationTrait for Migration {
    async fn up(&self, manager: &SchemaManager) -> Result<(), DbErr> {
        manager
            .create_table(
                Table::create()
                    .table(User::Table)
                    .col(ColumnDef::new(User::Id).uuid().primary_key())
                    .col(ColumnDef::new(User::OrgId).uuid().not_null())
                    .col(ColumnDef::new(User::Name).string().not_null())
                    .col(ColumnDef::new(User::Email).string().not_null().unique_key())
                    .to_owned(),
            )
            .await
    }

    async fn down(&self, manager: &SchemaManager) -> Result<(), DbErr> {
        manager.drop_table(Table::drop().table(User::Table).to_owned()).await
    }
}
```

Register it in the migrator:

```rust title="crates/migrations/src/lib.rs"
mod m20260601_000010_create_user;

pub use sea_orm_migration::prelude::*;

pub struct Migrator;

#[async_trait::async_trait]
impl MigratorTrait for Migrator {
    fn migrations() -> Vec<Box<dyn MigrationTrait>> {
        vec![Box::new(m20260601_000010_create_user::Migration)]
    }
}
```

The reference workspace adds an `org` table for the `belongs_to` from
page 2 — for the tutorial, the `org_id` column stays unconstrained
(every JWT carries an org id that just goes into the column).

## Run Postgres

The workspace already ships a Postgres in `compose.yml`. Bring it up
once.

```bash frame="terminal"
$ just db up
postgres ready on 127.0.0.1:5432

$ just db migrate
applied: m20260601_000010_create_user
```

`just db migrate` runs `Migrator::up` against
`NESTRS_DATABASE__URL` — the env var is set by `just db up`.

## Transactions wrap mutating routes automatically

`DatabaseModule` registers a request interceptor that classifies the
HTTP method and installs the right executor before the handler runs:

| HTTP method | Ambient executor |
|-------------|------------------|
| `GET` / `HEAD` / `OPTIONS` / `TRACE` | Pool — read-only, no transaction |
| `POST` / `PATCH` / `PUT` / `DELETE` | Transaction — committed on 2xx/3xx, rolled back otherwise |

The contract you write code against: every successful mutation
commits, every error response unwinds. No `tx.begin()` / `tx.commit()`
in the service.

## Run the binary

```bash frame="terminal"
$ NESTRS_DATABASE__URL=postgres://nestrs:nestrs@127.0.0.1:5432/nestrs \
    cargo run -p tutorial-api
INFO nest_rs::http: starting on 0.0.0.0:3000
INFO nest_rs::orm: connected to postgres
```

Hit the route with a token from the auth helper (the reference
`apps/platform-auth` issues one; for the tutorial, page 8 uses a
hand-signed test token from `nest-rs-testing`):

```bash frame="terminal"
$ curl -s -H "Authorization: Bearer $TOKEN" \
    -H 'Content-Type: application/json' \
    -d '{"name":"Ada","email":"ada@acme.test"}' \
    http://localhost:3000/users
{"id":"018f…","name":"Ada","email":"ada@acme.test"}

$ curl -s -H "Authorization: Bearer $TOKEN" \
    http://localhost:3000/users
[{"id":"018f…","name":"Ada","email":"ada@acme.test"}]
```

Now a failure path. A unique-violation on `email` returns `500`, and
nothing persists:

```bash frame="terminal"
$ curl -s -X POST -H "Authorization: Bearer $TOKEN" \
    -H 'Content-Type: application/json' \
    -d '{"name":"Ada","email":"ada@acme.test"}' \
    http://localhost:3000/users
HTTP/1.1 500 Internal Server Error

$ curl -s -H "Authorization: Bearer $TOKEN" http://localhost:3000/users
[{"id":"018f…","name":"Ada","email":"ada@acme.test"}]
```

One row, not two. The transaction unwound on the duplicate.

<Aside type="tip">
Set `RUST_LOG=nest_rs::orm=trace` to watch the `BEGIN` / `COMMIT` /
`ROLLBACK` pairs scroll past. The same `request_id` tags the
interceptor span and the SQL spans.
</Aside>

## What you have now

- A `DatabaseModule::for_root(None)` import in the app — the data
  layer is on.
- A migration adding the `user` table, applied through
  `just db migrate`.
- `POST /users` persists into Postgres inside an automatic transaction;
  a failing handler rolls back; the SQL is filtered by the caller's
  ability.
